* 管理节点和存储节点，即 NeonSAN 节点，所用系统盘容量均不可低于 480G，且为 SSD 介质。

* Zookeeper，MySQL Galera 属于 NeonSAN 管理服务，推荐部署在单独的物理节点上，独享硬件资源，避免与其他服务融合部署争抢资源降低集群稳定性。

* 在严肃场景中（DB 数据库、银行交易系统、客户重要机密数据），无论是 SANC 融合架构或 NeonSAN 独立架构，都建议使用独立物理节点部署管理服务。

* 常规业务场景中（OA、邮件、内外网办公），云平台 SANC 融合架构（节点小于 12 个），NeonSAN 管理服务部署推荐优先级：
+
(1) NeonSAN 管理服务优先部署在云平台 KS 管理节点虚机内，虚机配置：vCPU 不低于 8core，虚拟内存不低于 16G。
+
(2) 若存储节点有全闪硬件，NeonSAN 管理服务融合部署在全闪节点上。
+
(3) 若不满足 1、2，NeonSAN 管理服务融合部署在混插节点上。
+
(4) SANC 融合架构中存储节点数大于 12 台，推荐 SANC 分离架构，同时 NeonSAN 管理服务推荐部署在独立物理节点。

* 常规业务场景中（OA、邮件、内外网办公），云平台 SANC 分离架构，若节点数**大于 24**，管理服务推荐部署优先级：
+
(1) NeonSAN 管理服务优先部署在独立物理节点上。
+
(2) 若无独立管理节点，部署在云平台 KS 管理节点虚机内，虚机配置：vCPU 不低于 8core，虚拟内存不低于 16G。
+
(3) 若不满足 1 和 2，存储节点有全闪硬件，NeonSAN 管理服务融合部署在全闪节点上。
+
(4) 若不满足 1、2 和 3，NeonSAN 管理服务融合部署在混插节点上。

* 云平台 SANC 融合架构和 SANC 分离架构（节点数初期规划小于 24 后期扩容超过 24），NeonSAN 管理服务不管部署在 KS 节点的虚机内还是与存储节点融合部署，在客户条件允许的前提下，推荐将管理服务迁移到独立物理节点，迁移方案参见link:https://box.yunify.com/s/6D3XXjScJEwGLTa[《NeonSAN 管理服务迁移指南》]。若客户环境条件不允许将管理节点迁移至独立物理节点上，则可采用其他方案。

* NeonSAN 3 台独立管理节点支持管理 3 ~ 4096 个存储节点，最佳实践为 200 个存储节点。

* 独立管理节点硬件配置推荐：CPU Intel 4210*2，内存 4*16G，系统盘 2*480G SSD 网卡 Mellanox CX4  10GB 双口万兆带模块，4 个千兆电口，冗余电源，36 个月 7*24*4 服务。

* Center，Store 节点融合部署，共享物理资源。NeonSAN 集群至少部署 3 个 center 服务，多于三个节点的集群，每个存储池需要有一个归属的 center，可按存储池数量部署相应数量的 center。若集群中的存储池数超过 5 个，则只需部署 5 个 center 即可。

* Monitor 监控服务与 NeonStore 服务融合部署于存储节点，且每个 Store 节点均需要部署 Monitor 服务以监控本节点的故障。

* Prometheus 存储监控数据并根据规则触发相应的告警。Prometheus 和 Zookeeper 或 Mysql 角色部署在一起，且一套 NeonSAN 集群中至少有 2 个节部署了 Prometheus 以做高可用。另外，若需要开启卷性能监控功能，则部署 Prometheus 的节点的系统盘大小不低于 480G，且需要将系统盘进行单独分区，且分区大小至少为 300G。

* AlertManager/qs-alert 和 Zookeeper 或 Mysql 角色部署在一起，且一套 NeonSAN 集群中至少有 2 个节点部署了 AlertManager 以做高可用。
