This section describes how to upgrade from KubeSphere Enterprise v3.5.0 to KubeSphere Enterprise v4.1.0 while keeping the current Kubernetes version unchanged.

== Prerequisites

* Contact the KubeSphere Enterprise support team to get the KubeSphere Enterprise v4.1.0 installation package and license.

* Ensure that the current version of KubeSphere Enterprise is v3.5.0. If your KubeSphere Enterprise version is lower than v3.5.0, please upgrade to v3.5.0 first.

* Ensure that the current Kubernetes version is v1.21.x, v1.22.x, v1.23.x, v1.24.x, v1.25.x, v1.26.x, v1.27.x or v1.28.x.

== Preparation

. Transfer the KubeSphere Enterprise v4.1.0 installation package to all nodes of the cluster.
+
[.admon.note,cols="a"]
|===
|Note

|
To know the purpose of each file in the installation package, refer to link:../../02-install-kubesphere/02-install-kubernetes-and-kubesphere/#_view_the_installation_package[View the Installation Package].
|===

. Use **backup.sh** to back up critical resources from all projects under the system workspace, and back up all custom roles and permission-related resources.
+
--
Execute the following command to start the backup. The relevant resources will be backed up to the **backup-$timestamp** directory.


[,bash]
----
bash backup.sh
----

The backup will be completed as shown in the following picture.

image:/images/ks-qkcp/zh/v4.1/backup.png[ backup, 100%]
--

. Contact the KubeSphere Enterprise support team to determine the upgrade parameters for each component based on configurations in **cc.yaml** of current cluster and the patch of KubeSphere extensions. Then, create the **ks-core-values.yaml** file. You can refer to the **ks-core-values.yaml** sample file in the installation package.
+
--
[.admon.attention,cols="a"]
|===
|Attention

|
* In `ks-core-values.yaml`, you can set the extensions that are not enabled in the cluster to `enabled: false`. The components `core` and `iam` must be configured as `enabled: true` and cannot be disabled.

* Some extensions, in certain circumstances (such as requiring access to external service addresses), cannot use **Default Configuration** during the upgrade and require manual configuration of some parameters. Refer to the **Component Upgrade Configuration Documentation** to check if your extensions have this issue and make the necessary changes in `ks-core-values.yaml`.
Please view the "Default Configuration" and "Component Upgrade Configuration Documentation" in the **Upgrade Configuration Documentation** folder in the KubeSphere Enterprise v4.1.0 user guide.

|===
--

. Copy the **charts** folder from the installation package and the created **ks-core-values.yaml** file to the **tools** directory where **upgrade.sh** is located.

. Run the **pre-check.sh** script from the installation package to check if the upgrade conditions are met. Ensure that all checks pass. If any check fails, the subsequent upgrade is likely to fail.

== Single Cluster Upgrade

. On the host cluster node, switch to the **tools** directory of the installation package (this directory should contain the **charts** folder, **ks-core-values.yaml**, and **upgrade.sh**).

. Execute the following command to start upgrading KubeSphere Enterprise.
+
--
[,bash]
----
bash upgrade.sh host
----
--

. Follow the steps below to check if the host cluster has been successfully upgraded.

.. Run the following command; the upgrade job should have completed successfully as shown below.
+
[,bash]
----
root@xxx:~# kubectl -n kubesphere-system get job | grep upgrade
ks-core-post-upgrade    1/1           16s        28m
ks-core-pre-upgrade     1/1           58s        31m
prepare-upgrade         1/1           26s        32m
----

.. Run the following command; if the output is as shown below, it indicates that the ks-core Helm Chart has been successfully deployed.
+
[,bash]
----
root@xxx:~# helm list -n kubesphere-system 
NAME    	NAMESPACE        	REVISION	UPDATED                                	STATUS  	CHART         	APP VERSION
ks-core 	kubesphere-system	6       	2024-04-09 14:54:09.349488995 +0800 CST	deployed	ks-core-1.0.0	v4.1.0
----

.. Run the following command; the pods (ks-apiserver, ks-console, ks-controller-manager) should be in the running state as shown below.
+
[,bash]
----
root@xxx:~# kubectl -n kubesphere-system get pod
NAME                                     READY   STATUS      RESTARTS        AGE
ks-apiserver-594bb884db-phrlv            1/1     Running     0               28m
ks-console-7cd55dd9f9-f2jl9              1/1     Running     0               28m
ks-controller-manager-785fc676dc-9d9j2   1/1     Running     0               28m
----

.. Run the following command; if the state of all extensions is **Installed**, it indicates that all extensions have been successfully installed.
+
[,bash]
----
root@xxx:~# kubectl get installplan
NAME            STATE
storage-utils   Installed
devops          Installed
network         Installed
...
----

.. After logging in to the KubeSphere Enterprise web console, click the **username** in the top right corner > **About**. The displayed platform version should be v4.1.0.

. Use the IP address of the original web console, administrator username, and password, and log in to the v4.1.0 {ks_product-en} web console using a web browser.

. Follow the activation prompts and click **Activate** to import the KubeSphere Enterprise license.

. Check if all functions and data of the host cluster are normal.

== Multi-cluster Upgrade

. Follow the steps in the link:#_single_cluster_upgrade[Single Cluster Upgrade] section to complete the upgrade of the host cluster.

. On the member cluster node, switch to the **tools** directory of the installation package (this directory should contain the **charts** folder, **ks-core-values.yaml**, and **upgrade.sh**).

. Execute the following command to start upgrading KubeSphere Enterprise.
+
[,bash]
----
bash upgrade.sh member
----

. Follow the steps below to check if the member cluster has been successfully upgraded.

.. Run the following command; the upgrade job should have completed successfully as shown below.
+
[,bash]
----
root@xxx:~# kubectl -n kubesphere-system get job | grep upgrade
ks-core-post-upgrade    1/1           16s        28m
ks-core-pre-upgrade     1/1           58s        31m
prepare-upgrade         1/1           26s        32m
----

.. Run the following command; if the output is as shown below, it indicates that the ks-core Helm Chart has been successfully deployed.
+
[,bash]
----
root@xxx:~# helm list -n kubesphere-system 
NAME    	NAMESPACE        	REVISION	UPDATED                                	STATUS  	CHART         	APP VERSION
ks-core 	kubesphere-system	6       	2024-04-09 14:54:09.349488995 +0800 CST	deployed	ks-core-1.0.0	v4.1.0
----

.. Run the following command; the ks-agent pod should be in the running state as shown below.
+
[,bash]
----
root@xxx:~# kubectl -n kubesphere-system get pod
NAME                          READY   STATUS      RESTARTS        AGE
ks-agent-5dc5b57977-4x6mf     2/2     Running     0               59m
----

. After the member cluster is successfully upgraded, execute the following command on the host cluster to modify the status of the member cluster so that the extension agent can be scheduled to the member cluster.
+
--
[,bash]
----
kubectl get clusters.cluster.kubesphere.io {member-x} -o json | jq 'del(.status.conditions[] | select(.type=="Schedulable"))' | kubectl apply -f -
----

[.admon.note,cols="a"]
|===
|Note

| Run `kubectl get cluster` to get cluster names and replace `{member-x}`.

|===

If the cluster node does not support the `jq` command, you can achieve this by editing the cluster resource. Edit the member cluster's `status.conditions` and set the `status` to `true` for the condition with `type: Schedulable`.

[,bash]
----
kubectl edit clusters.cluster.kubesphere.io {member-x}
----

[,yaml]
----
status:
  conditions:
  - lastTransitionTime: "2024-04-11T09:12:22Z"
    lastUpdateTime: "2024-04-11T09:12:22Z"
    message: Cluster has been initialized
    reason: Initialized
    status: "True"
    type: Initialized
  - lastTransitionTime: "2024-04-12T07:59:14Z"
    lastUpdateTime: "2024-04-12T07:59:14Z"
    message: Cluster has joined federation control plane successfully
    status: "True"
    type: Federated
  - lastTransitionTime: "2024-04-12T08:02:56Z"
    lastUpdateTime: "2024-04-12T08:02:56Z"
    message: KS Core is available now
    reason: KSCoreReady
    status: "True"
    type: KSCoreReady
  - lastTransitionTime: "2024-04-12T08:02:56Z"
    lastUpdateTime: "2024-04-12T08:02:56Z"
    message: Not schedulable now
    reason: Upgrading
    status: "False"   # Edit here
    type: Schedulable
  - lastTransitionTime: "2024-04-12T08:24:27Z"
    lastUpdateTime: "2024-04-12T08:24:27Z"
    message: Agent has connected to proxy successfully.
    status: "True"
    type: AgentAvailable
  - lastTransitionTime: "2024-04-12T08:57:40Z"
    lastUpdateTime: "2024-04-12T08:57:40Z"
    message: Cluster is available now
    reason: Ready
    status: "True"
    type: Ready
----
--

. Log in to the {ks_product-en} web console of the host cluster, navigate to the **Cluster Management** page, and check if all functions and data of this member cluster are normal.

. Continue with the above steps for the next member cluster until all clusters have been upgraded.

== Post-Upgrade Configuration

Some extensions, such as `KubeSphere Gateway`, `WhizardTelemetry Monitoring`, `WhizardTelemetry Alerting`, `WhizardTelemetry Notification`, `RadonDB DMP`, require additional configuration after the upgrade. Please refer to the **Component Upgrade Configuration Documentation** in the **Upgrade Configuration Documentation** folder of the KubeSphere Enterprise v4.1.0 user guide.

[.admon.attention,cols="a"]
|===
|Attention

|
Be sure to refer to the **Component Upgrade Configuration Documentation** to configure `KubeSphere Gateway` after the upgrade. Otherwise, gateway-related functions will not be available.
|===

== FAQ

=== Unable to operate a member cluster that has not been upgraded

Issue: After the host cluster is upgraded to v4.1.0, unable to view and operate member clusters that has not been upgraded in the KubeSphere Enterprise web console.

Reason: APIs are incompatible between v4.1.0 and v3.5.0.

Solution:

* Upgrade the member clusters as soon as possible.
* You can directly access them through the web console of the member clusters before upgrading the member clusters.


=== Unable to log in to the web console of member clusters

Issue: Before upgrading the member cluster, it is impossible to log in through the member cluster's web console, with the message "incorrect username or password."

Reason: KubeSphere Enterprise v3.x by default prohibits direct access to the member cluster's web console.

Solution:

. Execute the following command to add the following configuration.
+
--
[,bash]
----
kubectl -n kubesphere-system edit cm kubesphere-config
----

[,yaml]
----
authentication:
  oauthOptions:
    accessTokenMaxAge: 0
    clients:
    - name: kubesphere
      secret: kubesphere
      redirectURIs:
      - '*'
----
--

. Execute the following command to restart ks-apiserver, which allows logging in to the member cluster's web console.

+
[,bash]
----
kubectl -n kubesphere-system rollout restart deployment ks-apiserver
----

=== The upgrade script exits unexpectedly

Issue: During the upgrade process, the upgrade script abnormally exits, resulting in a failed upgrade.

Solution: Execute the following command to view the logs of upgrade jobs and troubleshoot the upgrade failure.


[,bash]
----
kubectl -n kubesphere-system get job/prepare-upgrade

kubectl -n kubesphere-system get pod |grep prepare-upgrade

# View the logs of upgrade jobs
kubectl -n kubesphere-system logs prepare-upgrade-gvc6p
----

[,bash]
----
kubectl -n kubesphere-system get job/ks-core-pre-upgrade

kubectl -n kubesphere-system get pod |grep ks-core-pre-upgrade

# View the logs of upgrade jobs
kubectl -n kubesphere-system logs ks-core-pre-upgrade-4n7tp
----

[,bash]
----
kubectl -n kubesphere-system get job/ks-core-post-upgrade

kubectl -n kubesphere-system get pod |grep ks-core-post-upgrade

# View the logs of upgrade jobs
kubectl -n kubesphere-system logs ks-core-post-upgrade-4nzhk
----

=== The permission items of custom roles are empty

Issue: After the upgrade, the permission items of custom roles are empty, causing users assigned with custom roles unable to use KubeSphere Enterprise normally.

Solution:

The upgrade script `backup.sh` has backed up custom roles and their bindings. However, due to changes in permissions, it can not automatically convert their permissions during the upgrade. Therefore, users need to manually supplement the permissions of custom roles after the upgrade.

. After backing up using `backup.sh`, check the already backed up custom roles in `backup-$timestamp/kse35-backup-iam-xxx.yaml`.

. Find `metadata.annotations[iam.kubesphere.io/aggregation-roles]`, and compare its values with the names in the **3.x** column of the table. Then, find the corresponding permission names in the **Description** column on the KubeSphere Enterprise web console.
+
--
[.admon.note,cols="a"]
|===
|Note

|
Please refer to the following **Platform** table for platform custom roles, the following **Workspace** table for workspace custom roles, and the following **Project** table for project custom roles, to confirm the corresponding relationships.
|===
--

**Example:**

The information of the backed-up platform custom role `test` is as follows:

```yaml
apiVersion: iam.kubesphere.io/v1alpha2
kind: GlobalRole
metadata:
  annotations:
    iam.kubesphere.io/aggregation-roles: '["role-template-view-roles","role-template-view-users","role-template-view-basic","role-template-manage-clusters","role-template-view-clusters"]'
    kubesphere.io/alias-name: test-name
    kubesphere.io/creator: admin
  creationTimestamp: "2024-04-11T11:59:08Z"
  generation: 3
  name: test     # The name of the custom role
  resourceVersion: "529194"
  uid: a3ad5e39-2959-4e14-aa8f-e0c0ad8e6889
rules:
    ...
    ...
``` 

. Get the value of `metadata.annotations[iam.kubesphere.io/aggregation-roles]` as follows:
+
```yaml
'["role-template-view-roles","role-template-view-users","role-template-view-basic","role-template-manage-clusters","role-template-view-clusters"]'
```

. Compare the values of `metadata.annotations[iam.kubesphere.io/aggregation-roles]` with the **3.x** column in the following **Platform** table. The corresponding permission items in the **Description** column of the table should be checked in the web console, including **Role Viewing**, **User Viewing**, **Basic Viewing**, **Cluster Management**, and **Cluster Viewing**. However, since **Basic Viewing** is marked as `none` in the **4.1** column, **Basic Viewing** cannot be selected.

. Edit the permissions of the `test` role on the **Platform Roles** page in the KubeSphere Enterprise web console.
+
--
// image:/images/ks-qkcp/zh/v4.1/edit-permission.png[ edit-permission, 100%]

Select **Role Viewing**, **User Viewing**, **Cluster Viewing**, and **Cluster Management**.

// image:/images/ks-qkcp/zh/v4.1/check-permission.png[ check-permission, 100%]
--

[.admon.attention,cols="a"]
|===
|Attention

|
Some permission items have been removed in v4.1.0 and are marked as `none`. Others have not been removed, but for security reasons, they cannot be granted to custom roles and are marked as `can not use`. Permission items marked as `none` or `can not use` cannot be selected when editing the permissions of custom roles.
