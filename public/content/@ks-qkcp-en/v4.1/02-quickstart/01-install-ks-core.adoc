This section explains how to install Kubernetes and KubeSphere Enterprise.

The installation process will use the open-source tool KubeKey. For more information about KubeKey, please visit the link:https://github.com/kubesphere/kubekey[GitHub KubeKey repository].


== Prerequisites

* Contact the KubeSphere Enterprise support team to obtain the KubeSphere Enterprise v4.1.0 installation package.

* Prepare at least 1 Linux server as a cluster node. In a production environment, to ensure high availability of the cluster, it is recommended to prepare at least 5 Linux servers, with 3 servers as control plane nodes and 2 servers as worker nodes. If you are installing KubeSphere Enterprise on multiple Linux servers, make sure all servers belong to the same subnet.

* The operating system and version of the cluster nodes must be Ubuntu 16.04, Ubuntu 18.04, Ubuntu 20.04, Ubuntu 22.04, Debian 9, Debian 10, CentOS 7, CentOS Stream, RHEL 7, RHEL 8, SLES 15, or openSUSE Leap 15. The operating systems of multiple servers can be different. For support of other operating systems and versions, please contact KubeSphere Enterprise technical support.

* In a production environment, to ensure the cluster has sufficient computing and storage resources, it is recommended that each cluster node be configured with at least 8 CPU cores, 16 GB of memory, and 200 GB of disk space. In addition, it is recommended to mount an additional 200 GB of disk space in the **/var/lib/docker** (for Docker) or **/var/lib/containerd** (for containerd) directory of each cluster node for storing container runtime data.

* In addition to the cluster nodes, you also need to prepare a Linux server to create a image registry, and this server must be connected to the KubeSphere Enterprise cluster node network, with at least 100 GB of disk space mounted in the **/mnt/registry** directory.

* In a production environment, it is recommended to configure high availability for the KubeSphere Enterprise cluster in advance to avoid service interruption in the event of a single control plane node failure. For more information, please refer to the "Configure High Availability" section in the KubeSphere Enterprise Installation Guide.
+
--
// Note
include::../../../_ks_components-en/admonitions/note.adoc[]

If you plan to have multiple control plane nodes, be sure to configure high availability for the cluster in advance.

include::../../../_ks_components-en/admonitions/admonEnd.adoc[]

* By default, KubeSphere Enterprise uses the local disk space of the cluster nodes as persistent storage. In a production environment, it is recommended to configure an external storage system as persistent storage in advance. For more information, please refer to the "Configure External Persistent Storage" section in the KubeSphere Enterprise Installation Guide.

* If the cluster nodes do not have a container runtime installed, the installation tool KubeKey will automatically install Docker as the container runtime for each cluster node during the installation process. You can also manually install containerd, CRI-O, or iSula as the container runtime in advance.
+
--
// Note

CRI-O and iSula have not been fully tested for compatibility with KubeSphere Enterprise, and there may be unknown issues.

* Make sure the DNS server addresses configured in the **/etc/resolv.conf** file on all cluster nodes are available. Otherwise, the KubeSphere Enterprise cluster may encounter domain name resolution issues.

* Make sure you can use the **sudo**, **curl**, and **openssl** commands on all cluster nodes.

* Make sure the time is synchronized on all cluster nodes.

== Install Dependencies

You need to install socat, conntrack, ebtables, and ipset for all cluster nodes. If the above dependencies already exist on each cluster node, you can skip this step.

On Ubuntu systems, run the following command to install the dependencies on the servers:

// Bash
include::../../../_ks_components-en/code/bash.adoc[]

sudo apt install socat conntrack ebtables ipset -y

include::../../../_ks_components-en/code/codeEnd.adoc[]

If the cluster nodes use other operating systems, replace **apt** with the corresponding package management tool for the operating system.


== View the Installation Package

Have a view of the KubeSphere Enterprise installation package to proceed with the following steps.

The installation package contains the following files:

[,bash]
----
kse-all-v4.1.0-offline-linux-amd64/
├── charts
│   ├── ks-core                   # KubeSphere Enterprise core components
│   ├── kse-extensions-publish    # For publishing KubeSphere Enterprise extensions
│   ├── csi-qingcloud             # For integrating with QingCloud cloud storage
│   ├── csi-neonsan               # For integrating with NeonSan storage
│   └── nfs-client-provisioner    # For integrating with NFS storage
├── tools
│   ├── pre-check.sh              # For cluster environment check before KubeSphere Enterprise upgrade
│   ├── backup.sh                 # For backup of critical resources in the system workspace before KubeSphere Enterprise upgrade
│   └── upgrade.sh                # KubeSphere Enterprise upgrade script, allowing fast upgrade
│   ├── oras                      # OCI tool, for image synchronization operations
│   └── ks-core-values.yaml       # The example configuration file, used to configure upgrade parameters for each component
├── kse-extensions                # Containing installplans for all extensions, used for quick installation of KubeSphere Enterprise extensions
├── config-sample.yaml            # Template for the installation configuration file
├── create_project_harbor.sh      # For quickly creating a harbor project
├── kk                            # Cluster deployment tool
├── kubekey-artifact.tar.gz       # KubeSphere Enterprise artifact, containing the binaries and images required for cluster deployment
└── manifest-v410-amd64.yaml      # KubeSphere Enterprise artifact manifest, containing the versions and image lists of each component
----

== Create a Private Image Registry

[.admon.attention,cols="a"]
|===
|Attention

|
If you already have a image registry available, you can skip this step. But you need to replace the default address of the private image registry **dockerhub.kubekey.local/kse** with your actual image registry address.
|===

. Set the information of the server used to create the private image registry under the **spec:hosts** parameter in **config-sample.yaml**.
+
--
[,yaml]
----
spec:
  hosts:
  - {name: registry, address: 192.168.0.6, internalAddress: 192.168.0.6, user: ubuntu, password: Testing123}
----

include::../../_custom-en/installationAndUpgrade/installationAndUpgrade-para-hosts.adoc[]
--

. Set the server name for creating a private image registry under the **spec:roleGroups:registry** parameter (replace `<registry name>` with the actual name of the server set under the **spec:hosts** parameter).
+
[,yaml]
----
spec:
  roleGroups:
    registry:
    - <registry name>
----

. Set the **spec:registry:privateRegistry** parameter to the default address of the private image registry **dockerhub.kubekey.local/kse**, and save the file.
+
[,yaml]
----
spec:
  registry:
    registryMirrors: []
    insecureRegistries: []
    privateRegistry: dockerhub.kubekey.local/kse
----

. Execute the following command to initialize the private image registry.
+
====
[,bash]
----
./kk init registry -f config-sample.yaml -a kubekey-artifact.tar.gz
----

== Install Kubernetes

include::../../../_ks_components-en/admonitions/note.adoc[]


* If you already have an existing Kubernetes cluster, you can skip this step.

* The installation package includes dependencies for CentOS 7, Ubuntu 18.04, Ubuntu 20.04, and Ubuntu 22.04. If you are using one of these operating systems, you can use KK to automatically install the system dependencies by adding the `--with-packages` option to the installation command. If you want to integrate with other storage solutions, configure the relevant storage plugins under the **spec:addons** parameter in **config-sample.yaml ** or install them after the Kubernetes deployment.

== Install KubeSphere Enterprise

=== Step 1: Deploy ks-core

. On the cluster nodes, execute the following command to install KubeSphere Core.
+
====
[,bash]
----
helm upgrade --install -n kubesphere-system --create-namespace ks-core charts/ks-core \
--debug \
--wait \
--set cloud.enabled=false \
--set upgrade.enabled=false \
--set global.imageRegistry=dockerhub.kubekey.local/kse \
--set extension.imageRegistry=dockerhub.kubekey.local/kse
----

[.admon.attention,cols="a"]
|===
|Attention

|
Replace the default address **dockerhub.kubekey.local/kse** for **global.imageRegistry** and **extension.imageRegistry** with your actual image registry address.
|===

Depending on your hardware and network environment, you may need to configure traffic forwarding rules and open port 30880 in the firewall. If you see the following information, it means that ks-core installation is successful:


[,yaml]
----
NOTES:
Thank you for choosing KubeSphere Helm Chart.

Please be patient and wait for several seconds for the KubeSphere deployment to complete.

1. Wait for Deployment Completion

    Confirm that all KubeSphere components are running by executing the following command:

    kubectl get pods -n kubesphere-system

2. Access the KubeSphere Console

    Once the deployment is complete, you can access the KubeSphere console using the following URL:

    http://192.168.6.10:30880

3. Login to KubeSphere Console

    Use the following credentials to log in:

    Account: admin
    Password: P@88w0rd

NOTE: It is highly recommended to change the default password immediately after the first login.

For additional information and details, please visit https://kubesphere.io.
----
====

. From the successful information, retrieve the **Console**, **Account**, and **Password** parameters to obtain the IP address of the KubeSphere Enterprise web console, the admin username, and the admin password. Use a web browser to log in to the KubeSphere Enterprise web console.

[.admon.note,cols="a"]
|===
|Note

|
By now, the {ks_product-en} web console only provides the core functionalities of KubeSphere Enterprise. To use the extensions, you need to continue with the following two steps to install extensions.
|===

=== Step 2: Publish Extensions

Execute the following command to import all extensions provided by KubeSphere Enterprise into the Extensions Center of the web console.

[,bash]
----
helm template -n kubesphere-system charts/kse-extensions-publish --set museum.enabled=true,global.imageRegistry=dockerhub.kubekey.local/kse | kubectl apply -f -
----

If you see the following information, it means that the extensions have been successfully published.

image:/images/ks-qkcp/zh/v4.1/verify-extension-release.png[verify-extension-release, 100%]

=== Step 3: Deploy Extensions

After publishing the extensions, log in to the KubeSphere Enterprise web console. You can view and install the extensions in the Extensions Center. Alternatively, follow the steps below to quickly deploy the extensions.

// . To install observability extensions, you should first deploy vector (WhizardTelemetry Data Pipeline) and whizard-monitoring (WhizardTelemetry Monitoring). Otherwise, you can skip this step.
// +
// --
// [,bash]
// ----
// # Install vector and whizard-monitoring
// kubectl apply -f kse-extensions/vector-installplan.yaml 
// kubectl apply -f kse-extensions/whizard-monitoring-installplan.yaml

// # Check the installation state, and wait for the state to change to Installed
// kubectl get installplan vector -w
// kubectl get installplan whizard-monitoring -w
// ----
// --

. Execute the following commands to install specific extensions or all extensions.
+
--  
[,bash]  
----  
# To install a specific extension, please replace {EXTENSION-NAME} with the actual name.
kubectl apply -f kse-extensions/{EXTENSION-NAME}-installplan.yaml  
  
# To install all extensions  
kubectl apply -f kse-extensions  
----

[.admon.note,cols="a"]
|===
|Note

|
* Get `{EXTENSION-NAME}` in the `kse-extensions` directory of the KubeSphere Enterprise installation package, or refer to the picture below.
|===

[.admon.attention,cols="a"]
|===
|Attention

|
For extensions that have extension configurations modified, please do not re-execute `kubectl apply -f kse-extensions/{EXTENSION-NAME}-installplan.yaml` to avoid overwriting the configuration of the extensions.
|===
--

. Execute the following command and wait for the state of the extensions to change to **Installed**, which indicates a successful deployment.
+
--
[,bash]
----
kubectl get installplan -w
----
