企业云平台提供对象存储服务。对象存储通过与云平台 IaaS 对接，能够直接通过云平台控制台进行访问与使用，为用户或企业提供了一个无限容量的在线文件存储和访问平台。对象存储作为企业的数据存储和流转中心，用户可通过浏览器、HTTP RESTful API 、S3 API、SDK 和 FTP 等方式高效存取和管理文件，从而支撑企业丰富的上层业务和数据分析系统使用。

== 功能特性

* 存储空间管理：基于云平台多租⼾提供 Bucket ⽣命周期管理，⽀持多租⼾隔离。
* 文件管理：支持上传、下载、分享文件。支持设置相应的生命周期规则，定期将非热门数据转换为低频访问、归档存储或冷归档存储，并删除过期数据。
* 数据访问权限控制：支持 Bucket ACL、Bucket Policy、CORS。
** Bucket ACL：用户根据需求将存储空间配置为公开可读或公开可写（即不附带认证信息的匿名访问），也可针对特定用户配置读写权限。
** Bucket Policy：存储空间策略可以通过细致地指定针对 API 级别的控制，来实现 Bucket ACL 和 Object ACL 所不能实现的一些功能，比如防盗链。
** CORS：支持通过为 Bucket 配置 CORS 规则实现跨源资源共享。当用户利用 JavaScript AJAX 向对象存储发起的请求属于跨源请求时，默认情况下浏览器出于安全考虑，不允许调用不同域名下的资源。
* 数据分析处理：支持图片处理与鉴黄。
* 数据加密：支持由用户提供密钥的加密方式，服务端通过用户提供的密钥对上传的对象进行加密处理。
* Bucket 跨区域复制：允许用户开启跨区域（数据中心）之间的 Bucket 自动、异步的复制 Object，可应用于数据备份于容灾、合规性要求等场景。
* 静态网站托管：面向静态网页内容提供托管服务，对静态网站的文件存储、访问权限控制、CDN 分发加速以及安全保障等一站式解决方案。

== 全局数据模型

下图为对象存储的全局数据模型，可理解为一个逻辑上的视图。

image::/images/ep_offline/ep_twp/qingstore_01.png[,50%]

对象存储在 Global 级别 和 Zone 级别都有相应的管理服务。Global表示全局， 由多个 Zone 组成。例如，在北京区域部署了一套对象存储，同时在上海数据中心也部署了一套对象存储，这两套对象存储属于同一个 Global。

Zone 由很多个 Bucket（存储桶）组成，在使用对象存储时，必须要先申请一个存储桶，然后才能向存储桶中上传对象数据，以存储桶为单位来存储和管理对象数据。

在同一个 Zone 下面可以有很多个存储桶，存储桶里有各种各样的对象数据，对象数据对类型大小没有限制，单个存储桶中对象的数量也没有限制，可以无限量上传对象数据。

== 系统架构设计 

将对象存储的后台系统进行简化和抽象后，可以得到如下架构图。

image::/images/ep_offline/ep_twp/qingstore_02.png[,50%]

对象存储主要有接入子系统、索引子系统、存储子系统和事件子系统几个模块构成。

* 接入子系统
+
对象存储通过 Restful 提供在线访问，接入子系统相当于在线服务的后台，负责接收请求、解析协议等工作。
+
在接入子系统中最重要是 Gateway 服务，Gateway 服务本质上是在后台运行的 Server，它运行在网关节点上，主要功能是接收上游发下来的请求，做协议解析以及数据处理与数据读写，对象存储中最重要的数据读写功能基本都由它完成。
+
用户开始访问对象存储之前，会先访问 DNS 服务器，通过对象存储服务的域名拿到一个虚拟 IP 地址，这个虚拟 IP 会指向某一台网关节点，如果这台节点故障了，虚拟 IP 会自动迁移到另外存活的节点上。在请求到达节点后，对象存储会进一步做负载均衡，把请求分到多个节点的 Gateway 服务上。当请求转到一个 Gateway 上，若发现这个 Gateway 故障，系统自动会将请求重新转发，这就是请求级别的 Failover。通过这两个机制，对象存储可以保证访问请求都可以得到响应。

* 索引子系统
+
索引子系统用来存储和管理对象的元数据，元数据指的是对象的 Meta 信息，包括 Object 类型、大小、写入时间之类的信息。
+
索引子系统通过数据分片的方式来应对海量数据，把数据按切片存储。切片的意思是按照对象名称字母序进行切片，比如：第一个节点存 A-F 对象，第二个节点存 U-Z 对象，每一个节点会负责一个区间的数据，然后使用一个协调服务记录节点及其所负责的区域对应关系。如果 A-F 的对象数据太多，系统会把它拆成两个，增加一个节点进来，将 A-F 拆分为 A-D，A-D 保留在原有的节点，D-F 放在新的节点。通过切片存储，可以很方便地横向扩展，也提升了处理读写的能力。添加新的存储节点通过协调服务进行，数据的拆分和再平衡过程是完全自动进行的。
+
对象存储在单个节点上存储元数据时使用 KV 存储引擎，KV 存储引擎的索引结构是 LSM，LSM 索引结构最大的优势是写入非常快，可以提升系统的整体写入性能。
+
总体来说，分片加协调服务的方式使其有很好的扩展性，同时通过副本保障数据的安全。此外，在单个节点上采用 KV 存储引擎提升写入效率，同时支持 list 接口的高效查询。

* 存储子系统
+
存储子系统负责存储和管理数据实体本身，保证数据的可靠持久化存储。
+
对象存储在统一命名存储空间下将存储分成一个个存储组，每个存储组由各自的分布式文件系统组成。在单个存储组内，采用三副本进行存储，每次写入数据，等到三副本写入全部完成才会返回写入成功，保证数据的强一致性和安全性。
+
对象存储的数据传输采用 RDMA 高效传输协议，RDMA 是一种在不同的节点之间传输数据的机制，它不需要经过 CPU，直接通过硬件控制将数据从一个节点拷贝到另一个节点，和 CPU 的执行是并行的，是一种高效传输数据的方式。

* 事件子系统
+
事件子系统在对象存储的架构中，不处于数据读写的核心流程上，但它提供了很多非常重要的功能，如生命周期管理、跨区域复制以及自定义回调都是基于事件子系统进行设计的。
+
事件子系统的基本逻辑便是产生事件和处理事件，此处的事件可以理解成消息系统。
+
产生消息有两条链路，一条链路是用户完成一条数据的读、写入、删除或者更新，索引子系统会产生事件，发送到事件子系统中；还有一条链路是用户直接通过接入子系统提交一个事件，可以进入事件子系统中。
+
事件子系统有很多预置的消费者进程，消费者会处理事件，这些事件是通过分布式消息队列保存的，消费者按照预设的逻辑读取事件并对其进行处理。
