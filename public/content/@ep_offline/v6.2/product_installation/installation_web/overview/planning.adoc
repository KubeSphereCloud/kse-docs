实施人员首先需根据用户的实际需求，确定待部署环境的规模、部署架构以及网络架构，并收集相关信息。

== 环境规模

实施人员可根据用户的实际业务量，评估待部署环境所需物理服务器的数量，确认待部署环境的规模。不同部署规模下，{product_name}对各节点服务器的要求，存在差异。具体内容，不在本文赘述。

== 部署架构

{product_name}的部署架构较为复杂，可分为云平台的部署架构与计算存储的部署架构。

=== 云平台部署架构

{product_name}根据管理服务与计算服务是否分离，可以分为标准部署与融合部署两种方式。

* **标准部署模式**
+
我们又称之为分离部署。该部署模式中，云平台的管理服务与计算服务分别部署于不同的物理服务器。

* **融合部署模式**
+
融合部署模式下，云平台的管理服务与计算服务共用服务器节点，部署于相同的物理服务器。


=== 计算存储部署架构

{product_name}使用 NeonSAN 作为存储时，我们称之为 Hyper-Sanc 架构；对接第三方存储时，我们称之为 Hyper-Tpsc 架构。

对于 Hyper-Tpsc 的部署架构，{product_name}目前仅支持分离部署的方式，即第三方存储服务使用专用的集中式存储控制器，不与{product_name}服务器共用物理节点。而使用 NeonSAN 作存储时，支持融合部署与分离部署。

* **Hyper-Sanc 独立部署模式**
+
是指 NeonSAN 的存储服务，不使用{product_name}的物理服务器，而是使用独立的物理服务器进行部署。
+

+
是指 NeonSAN 的存储服务与云平台的计算服务，共用物理服务器，即 NeonSAN 的相关服务，部署于云平台的 Hyper 节点上。

== 网络架构

{product_name}最新的实施方案不推荐部署 VG，如需部署 EIP 请使用 H3C-VG 或 SDN3.0 的 No VG 方案。若选用 H3C-VG 或 SDN3.0 的 No VG 方案时，云平台部署架构只能选择 `标准部署`。


目前支持以下三种网络模式。

* **SDN 3.0 网络模式**
+
使用 3.0 版本的 SDN 软件定义{product_name}网络。

* **SDN 2.0 网络模式**
+
使用 2.0 版本的 SDN 软件定义{product_name}网络。

* **VLAN 网络模式**
+
使用物理交换机的 VLAN 来划分{product_name}网络。
