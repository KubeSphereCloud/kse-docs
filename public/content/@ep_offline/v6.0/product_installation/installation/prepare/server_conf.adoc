== RAID 配置

在进行操作系统安装之前，我们需对各节点服务器上的磁盘进行配置操作。若当前环境为全闪环境，系统盘与数据盘使用的都是 SSD 盘，则可跳过该章节内容，直接进行操作系统的安装。若使用了机械盘，则需按照本章节内容对机械盘进行 RAID 配置。

另外，硬件 RAID 是否自带缓存对机械盘性能影响较大，因此，对 `RAID50` 与 `RAID1` 做如下要求：

* RAID50：必须使用带缓存的硬件 RAID。
* RAID1：硬件 RAID 可以不带缓存。

为提高缓存的利用率，需要缓存空间尽可能的大。所以在做 CAS 设备的时候，要避免将一块 SSD 磁盘分成多个区使用。正确的做法，是将整块 SSD 磁盘做成共享的缓存设备，再分给两个 drbd 使用。


磁盘做 RAID 的过程，不在这里描述，这里仅列举说明各节点的 RAID 配置。

=== KS 节点

KS 节点的数据盘需要做成一个 RAID。在数据盘足够的情况下，建议做 `RAID5` 或 `RAID50`。

[cols="1,2,2"]
|===
| 磁盘用途	| 存储介质	| RAID

| 系统盘	| -	| RAID1
| 数据盘    | SAS 盘 | RAID5/RAID50
|===


=== Hyper 节点

Hyper 节点上，系统盘均配置 `RAID1`。数据盘与缓存盘则根据存储类型与存储介质的不同，RAID 配置要求稍有区别，详细内容如下。

// * 数据盘
// +
// 当数据盘使用 NVMe 或 SSD 磁盘时，可使用直通模式，无需做 RAID。
// 当数据盘使用 NVMe 或 SSD 磁盘时，可使用直通模式，无需做 RAID。

如 SDS1.0 场景下，缓存盘使用 NVMe 时，则采用直通模式，若使用 SSD 时，则采用 RAID1 模式。

[cols='1,1,1,2']
|===
| 存储类型	| 磁盘用途  |  存储介质 | RAID

.3+| SDS1.0
| 数据盘| SAS 盘	| RAID50/RAID5
.2+| 缓存盘
| NVMe	| -
| SSD	| RAID1

.2+| SDS2.0

| 数据盘| NVMe	| -
| 数据盘| SSD	| -

.2+| Sanc-HDD

| 数据盘| SAS	| NeonSAN 混插场景默认采用数据盘直通模式。如果有多重数据冗余等需求可以做 RAID5/RAID50，要求单个 RAID 组容量不大于 50TB。
| 缓存盘| NVMe	| -

.2+| Sanc-SSD
| 数据盘 | NVMe	| -
| 数据盘	| SSD	| -

|===

== 配置 Hostname

操作系统成功安装完成后，用户需逐一登录各服务器，修改配置主机名。为方便管理与后期运维，企业云平台要求主机名称中应包含待部署环境的 Zone ID、逻辑机柜号以及逻辑机位号。

**主机名格式**

[source,shell]
----
<Zone ID><逻辑机柜号><逻辑机位号>

----

**示例**

待部署环境的 Zone ID 为 `testcloud1`，当前主机所在的逻辑机柜号为 `r01`，逻辑机位号为 `n00`。那么，对应的主机名应修改为：

[source,shell]
----
testcloud1r01n00

----

**操作步骤**

. 逐一登录待部署环境的各节点服务器。

. 执行以下操作，修改主机名。主机名需遵循前文规定的格式。
* 方式一：使用命令行进行修改，其中 `<hostname>` 为待使用的主机名，
+
[source,shell]
----
# hostnamectl set-hostname <hostname>

----
* 方式二：直接修改 `/etc/hostname`，写入待使用的主机名。
+
[source,shell]
----
# vi /etc/hostname
----

== 配置管理网络

操作系统安装完成，且主机名成功修改后，用户需要根据环境规划，依次登录每台服务器，为服务器配置管理网络。


. 依次登录所有服务器节点。
. 执行以下命令，修改 /etc/network/interfaces 配置。不同网络模式下，配置文件的内容填写，稍有区别。详情可参考 link:#_sdn_模式网卡配置[SDN 模式网卡配置]与 link:#_vlan_模式网卡配置[VLAN 模式网卡配置]。
+
[source,shell]
----
# vi /etc/network/interfaces
----

=== 网卡配置要求

企业云平台要求将同一网卡的两口做 Bond，以避免网卡的交叉使用。除此之外，还有如下要求：

* 对于 VG 节点，需要配置两对 Bond，一对用于连接管理网络，一对用于连接公网。
* 对于 Hyper-Sanc 的融合节点，配置一对 Bond 用于连接物理网络。另外两条私网线路配置存储私网。


=== SDN 模式网卡配置

若选择 SDN 网络模式，请参照如下配置说明进行修改。

[source,shell]
----
# This file describes the network interfaces available on your system
# and how to activate them. For more information, see interfaces(5).

# The loopback network interface
auto lo
iface lo inet loopback

auto ens2f0                  # 同一张网卡的一个网口名。
iface ens2f0 inet manual
bond-master bond0

auto ens7f0                 # 同一张网卡的另一个网口名。
iface ens7f0 inet manual
bond-master bond0

# mgmt network
auto bond0
iface bond0 inet static
address 10.10.0.11        # 需根据前期规划的管理网络进行填写。
netmask 255.255.255.0
gateway 10.10.0.254       # 需根据前期规划的网关节点 IP 进行填写。
bond-mode 4
bond-miimon 100
bond-lacp-rate 1
bond-slaves ens2f0 ens7f0  # 两网口做 Bond。
----


=== VLAN 模式网卡配置

若选择 VLAN 网络模式，请参照如下配置说明进行修改。

[source,shell]
----
# This file describes the network interfaces available on your system
# and how to activate them. For more information, see interfaces(5).

# The loopback network interface
auto lo
iface lo inet loopback

auto enp175s0f0                   # 同一张网卡的一个网口名。
iface enp175s0f0 inet manual
bond-master bond0

auto enp175s0f1                   # 同一张网卡的另一个网口名。
iface enp175s0f1 inet manual
bond-master bond0

auto bond0
iface bond0 inet manual
bond-mode 4
bond-miimon 100
bond-lacp-rate 1
bond-slaves enp175s0f0 enp175s0f1  # 两网口做 Bond。

# mgmt network

auto vlan11                        # vlan11 为 VLAN ID，应与交换机上规划的 VLAN ID 匹配。

iface vlan11 inet static
address 10.10.1.11                 # 需根据前期规划的管理网络进行填写。
netmask 255.255.255.0
gateway 10.10.1.254                # 需根据前期规划的网关节点 IP 进行填写。
vlan_raw_device bond0
----

== 配置系统时间

为确保操作系统能正常运行，我们需要对服务器进行时间同步操作。

**注意事项**

* 若待部署环境已经规划了单独的 NTP 服务器，则无需执行此步骤，只需要在配置文件中填写 NTP 服务器地址即可。
* 若待部署环境没有规划单独的 NTP 服务器，则在 Installer 部署过程中，系统会自动选择 VG 节点或 KS 节点作为时钟同步的服务器，并对其他的物理服务器和管理虚拟机做时间同步。因此，为确保环境时间的准确性，用户需手动配置 VG 节点或 KS 节点的时间。

**操作步骤**

. 根据前期规划内容，逐个登录 VG 或 KS 节点。

. 执行以下命令，查看服务器当前系统时间。
+
[source,shell]
----
# date
----
. 若系统时间不正确，与实际时间有偏差，执行以下命令，修改服务器时间。
+
[source,shell]
----
# date -s “yyyymmdd hh:mm:ss”
----
+
如修改系统时间为 `20220831 12:00:00`，则命令行为：
+
[source,shell]
----
# date -s “20220831 12:00:00”
----
. 时间修改后，执行以下命令，保存修改。
+
[source,shell]
----
# hwclock --systohc
----

== 校验服务器信息

以上操作执行完成后，需对每个节点服务器进行重启，检验以上配置修改是否成功。

. 重启服务器。
** 方式一：通过服务器管理页面进行重启。我们推荐该方式。
** 方式二：登录节点服务器，执行 `reboot` 命令重启。
** 方式三：直接对服务器进行下电再上电操作。

. 待各服务器重启完成后，检查如下信息。
** 主机名称配置是否正确。
** 网络配置信息是否正确。
** 硬盘盘符型号识别是否正确。
** VG 或 KS 节点时间是否正确。
