DevOps（Development 和 Operations 的组合词）是⼀种重视“软件开发⼈员（Dev）”和“IT 运维技术⼈员（Ops）”之间沟通合作的⽂化、运动或惯例。透过⾃动化“软件交付”和“架构变更”的流程，来使得构建、测试、发布软件能够更加地快捷、频繁和可靠，可以把 DevOps 看作开发（软件⼯程）、技术运营和质量保障（QA）三者的交集。传统的软件组织将开发、IT 运营和质量保障设为各⾃分离的部⻔，在这种环境下如何采⽤新的开发⽅法（例如敏捷软件开发），是⼀个重要的课题。按照从前的⼯作⽅式，开发和部署，不需要 IT⽀持或者 QA 深⼊的跨部⻔的⽀持；⽽现在却需要极其紧密的多部⻔协作。⽽ DevOps 考虑的还不⽌是软件部署，它是⼀套针对这⼏个部⻔间沟通与协作问题的流程和⽅法。

需要频繁交付的企业可能更需要对 DevOps 有⼀个⼤致的了解。Flickr 发展了⾃⼰的 DevOps 能⼒，使之能够⽀撑业务部⻔“每天部署 10 次”的要求──如果⼀个组织要⽣产⾯向多种⽤户、具备多样功能的应⽤程序，其部署周期必然会很短。这种能⼒也被称为持续部署。

DevOps 的核⼼理念就是 CI/CD （持续集成/持续部署）。在以往，CI/CD 需要和很多服务器以及服务器上的不同应⽤打交道，要花费⼤量⼈⼒和时间。尤其随着软件⽇趋复杂的情况下，微服务架构也逐渐流⾏起来。在这种情况下做 CI/CD 所需的成本更⼤。Docker 的出现改变了这⼀现状，Docker 的持续集成、版本控制、可移植性、隔离性和安全性所带来的敏捷和经济恰恰和 DevOps 的⽬标是⼀致的。

DevOps 项⽬是 KubeSphere 项⽬基础上的⼀个应⽤项⽬，为⽤户提供从开发到部署的⼀整套 AutoDevOps 流程。⽤户只要提供 Git 地址，就能为其⾃动⽣成⼀个以该源代码为基础的部署好的测试环境甚⾄是⽣产环境。

== 核心功能

1. 在 KubeSphere 上构建，原⽣⽀持 Kubernetes。
2. 基于 GitOps 理念，支持基于代码仓库运行流水线和持续部署应用。
3. 在 SCM 中增强了代码仓库管理，支持将 Kubernetes 资源同步至代码仓库。
4. 内建主流语⾔和主流框架的模板，降低编译--测试--部署的复杂性，⼤部分主流语⾔和框架需配置项较少。
5. 对于⾼级⽤户或者⾮主流框架，可通过⾃定义 Pipeline 的⽅式构建。
6. 项⽬架构微服务化，⼀键部署到主流容器服务中。该项⽬最终能够在各个⼚商的云平台和私有云快速部署，提⾼企业的开发效率。
7. 支持 source to image / binary to image，快速交付容器镜像。
8. 兼容 Jenkinsfile in & out of SCM（Source Code Management）两种模式。
9. 独立 DevOps 工程，提供访问可控、安全隔离的 CI/CD 操作空间。
10. 可视化流水线编辑工具，降低 CI/CD 学习成本。
11. 代码静态扫描。

== 架构设计

这些组件都是以微服务的形式提供。

1. API Server 系统核⼼组件，负责统⼀各个模块，管理 Pipeline 和提供与 KubeSphere 交互的接⼝。
