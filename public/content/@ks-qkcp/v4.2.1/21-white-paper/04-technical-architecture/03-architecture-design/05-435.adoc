== 核心功能

基于 Istio 微服务框架提供可视化的微服务治理功能，将 Kubernetes 的服务进行更细粒度的拆分。支持熔断、灰度发布、流量管控、限流、链路追踪等完善的微服务治理功能，支持代码无侵入的微服务治理。启用服务注册后，业务服务会增加一个 Proxy 代理容器。该代理容器会对流量数据进行转发记录等操作，通过 Jaeger 生成 TraceID，页面可通过该 ID 追溯流量整个生命周期，实现全链路监控。通过该代理容器，控制流量的比例及转发规则来实现实现灰度发布、流量限速、熔断等功能。

=== 架构设计

Istio 分为控制面和数据面，控制面主要由 Pilot 下发配置 xds 到 Proxy；数据面通过配置文件生成相应的路由，进行数据的转发。Proxy 使用 envoy 做为代理容器，与业务容器在同一个 Pod 中，使用 Iptables 规则进行流量拦截并转发。提供了丰富的 API，支持配置文件动态更新。

基于 Kubernetes 及 Istio，实现微服务治理功能：

* 熔断及限流：通过配置熔断规则，让请求达到某一个阈值时，终断流量或限制流量最大值。应用程序具备应对来自故障、潜在峰值和其他未知网络因素影响的能力。如设置 DestionationRules 的 maxConnections 或是 http1MaxPendingRequests，限制最大连接数或是 http 最大请求数量。
* 流量镜像：是一个以尽可能低的风险为生产带来变化的强大的功能。镜像会将实时流量的副本发送到镜像服务。镜像流量发生在主服务的关键请求路径之外。通过设置 VirtualService 智能路由中的镜像版本来实现此功能。
* 灰度发布：同时运行两个版本工作负载，通过 virtualService 的智能路由功能，控制流量到不同版本的比例，实现灰度发布。
* 流量监测：Envoy 提供了 Prometheus 的集成的多项指标，根据流量的采集值，绘制出流量治理拓扑图，能清楚体现服务之间调用关系，及实时流量。
* 链路追踪：Envoy 友好支持 Jaeger，根据 TraceId，可以很方便的追踪每个请求的路由状态，可以很直观地观察在哪一跳出现问题及延时。
* 负载均衡：通过 Istio DestinationRules，可以设置连接的策略，如：最小连接数、随机访问或轮巡。

image::/images/ks-qkcp/zh/technical-white-papers/435-3.png[]

基于 Spring Cloud 全家桶，服务之间通过 RPC 调用实现微服务治理功能。采用 Spring Cloud 后，不同业务可以拆分成不同的项目。采用微服务部署方式，服务提供者 Provider、消费者 Consumer、注册中心 Eureka Server、网关 Api Gateway(Zuul)，分别单独部署在托管容器集群服务运行时 K8s 的 Pods 里面，通过设置多副本或者 HPA 自动扩缩容，实现高可用。服务提供者、消费者通过 RPC 调用，通过注册中心实现服务发现。消费者通过负载均衡访问服务者，当流量达到一定阈值后提供熔断的功能（SpringCloud 内置组件 Ribbon 和 Hystrix）。路由 Zuul 是外网的入口，通过 URL 规则把请求分发到合适的服务，同时提供鉴权，过滤器功能，从而达到微服务治理目的。
