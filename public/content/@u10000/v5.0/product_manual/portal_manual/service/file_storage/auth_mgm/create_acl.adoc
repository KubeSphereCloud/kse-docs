== 操作步骤

用户自主创建的权限组，还需对其设置相应的 ACL 规则来满足其具体的访问权限控制需求。

//针对已创建的 NFS 权限组，用户还需创建相应的 link:#_acl_规则[ACL 规则]来满足其具体的权限控制需求。

. 成功登录 Portal。

. 点击**存储服务 > 文件存储 > 权限管理**，进入权限组列表页面。

. 在权限组列表页面，点击目标权限组，进入 ACL 列表页面。

. 在 ACL 列表页面，点击**创建 ACL**。

. 弹出**创建 ACL **对话框，用户根据页面提示信息填写相关参数后，点击**确定**。
+
可参考link:#_填写参数说明[填写参数说明]对页面参数进行填写。

. 用户可在 ACL 列表页面，查看已成功创建的 ACL 规则。
+
image::/images/u10000/stor_file_acl_list.png[stor_file_acl_list,100%]

== 填写参数说明

=== NFS 权限组

NFS 权限组的每一条 ACL 规则都是对不同 IP 或 IP 段客户端的权限限制。其匹配顺序由 ACL 的优先级决定。ACL 的优先级用数字 1-100 表示，数字越小，匹配优先级越高。

创建 NFS 权限组时，系统将默认创建一条 `*、none、不映射、优先级为 100` 的 ACL 规则，其表达的含义是：所有客户端均没有访问权限，此条 ACL 规则优先级最低。

相同优先级下，系统会按照 ACL 创建的先后顺序进行匹配，后创建的 ACL 优先。{product_name} 建议不同的 ACL 设置不同的优先级。

image::/images/u10000/stor_file_acl_create.png[stor_file_acl_create,40%]

**参数说明**

[cols="1,3a", options="header"]
|===
| 参数 |说明

|授权地址 |该 ACL 规则生效的 IP 或 IP 段。

|读写权限 | 针对**授权地址**中指定的客户端赋予的权限。

* `只读` 表示授权地址对文件系统拥有``只读``权限。
* `读写` 表示授权地址对文件系统拥有``读写``权限。
//* `无权限` 对应 None，授权地址对文件系统没有权限。授权地址为 `黑名单`，挂载时也会失败。

|映射关系 | 
* `不映射` 表示保留客户端的登录身份。
* `映射成 Nobody 用户` 表示将客户端的登录信息映射为匿名用户。可根据需求选择将``所有用户``或 ``Root 用户``映射为匿名用户。
** `所有用户` 表示将客户端的登录信息都映射为匿名用户。
** `Root 用户` 表示只有当客户端以 Root 身份登录文件系统时，把 Root 用户映射为匿名用户。若客户端使用其他用户身份登录，则保留登录身份。

|优先级 | * ACL 的优先级用数字 1-100 表示，数字越小，匹配优先级越高。

* ACL 的匹配顺序是由优先级决定的。

|===
